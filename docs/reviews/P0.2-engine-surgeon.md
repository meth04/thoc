## 2. Bệnh và cơ chế sửa

`run.py:259-261` (cũ) gọi `luu_checkpoint()` **rồi mới** `events.flush()` ⇒ offset ghi trước flush.
`run.py:206-224` (cũ) nạp checkpoint tick N nhưng **không đưa journal về tick N**.
`minds/transcript.py:51-52` (cũ) mở `"a"` với `self._n = 0` ⇒ `call_id` đếm lại từ 1.

Ba journal của `real60_spatial` dừng ở **ba tick khác nhau** sau cùng một lần kill (events→117,
llm_calls→118, transcript→119) ⇒ **không heuristic quét nội dung nào tìm được điểm cắt an toàn.**
Điểm cắt hợp lệ duy nhất là byte-offset ghi TẠI checkpoint, **sau flush+fsync**.

**Thứ tự mới (`engine/journal.py::RunJournals.checkpoint`):**

```
flush+fsync MỌI journal → capture(byte_offset, record_count, sha256_prefix)
  → journal_manifest.json (atomic) → pickle + checkpoint_moi_nhat.json → con trỏ journal (atomic)
```
Con trỏ resume ghi **cuối cùng** ⇒ crash giữa chừng ⇒ con trỏ vẫn ở checkpoint cũ hơn ⇒ nhất quán.

**Resume:** `verify` (size ≥ offset ∧ sha256(prefix) khớp ∧ identity khớp) → `restore`
(**MOVE** bytes tail sang `checkpoints/orphans/`, hash + index vào `recoveries[]`, rồi truncate)
→ `segment_id += 1` → writer mở với `start_seq`/`start_call_id` đúng.

**Accounting identity (`llm_calls.sqlite` — NGOẠI LỆ có chủ ý):**
```
call_burned (mọi row, KHÔNG BAO GIỜ DELETE)  =  call_effective (superseded=0)  +  rows_superseded
```
Row của đoạn bị bỏ **đã thực sự tốn token/quota/USD**; xóa chúng là làm đẹp chi phí (điều luật #6).
`run_meta.so_call` = `call_effective`; `run_meta.so_call_billed` = `call_burned`.

**5 journal (F-12):** `events.jsonl`, `transcript.jsonl`, `unrecognized_intents.jsonl` (Class A —
truncate+quarantine), `llm_calls.sqlite` (Class C — supersede), `metrics.jsonl` (Class B — dẫn xuất,
ghi đè cuối run từ pickle; **đã tự xác minh: không cần truncate**, chỉ assert tick duy nhất + liên tục,
`run.py::_ghi_metrics` fail-closed `E-JM-10`).

**`unrecognized_intents.jsonl` — sai lệch có chủ ý so với thiết kế:** model-architect §6.1 đề nghị thêm
`seq`/`seg`. Tôi **KHÔNG** làm: counter sẽ phải sống trong `World` (`engine/world.py:399-406` mở/đóng
file mỗi lần ghi), tạo đúng "hash-boundary hazard" mà D4 cấm. Nó vẫn vào manifest và vẫn được
truncate+quarantine — đó là cái F-12 thực sự cần.

## 6. Migration policy

- **Run cũ (không có `journal_manifest.json`) + `--resume` ⇒ FAIL-CLOSED (`E-JM-01`)**, in hướng dẫn.
  Escape hatch `--recover-journal`: quarantine **toàn bộ** journal → `replay_complete=false` +
  `artifact_status_forced=diagnostic_only_unreplayable` **vĩnh viễn**. Nó **KHÔNG BAO GIỜ** làm run xanh.
- Run cũ **không resume** (chỉ verify/replay/analyze): chạy bình thường. `journal_manifest_present`
  là WARN; `journal_continuity` (tính từ NỘI DUNG FILE) là HARD ⇒ run 1-phiên sạch vẫn PASS,
  run bẩn bị bắt (đúng như `mock60` PASS continuity còn `real60` FAIL).
- **Run MỚI ghi đè run-dir đã có journal** (vd `tools/verify_local.py` chạy lại cùng `--run-name`):
  journal cũ được **quarantine** vào `checkpoints/orphans/pre_run_<utc>/` + 1 dòng
  `journal_recovery.jsonl` kind `fresh_run_reset`, rồi bắt đầu từ 0. Trước P0.2 nó **append chồng**
  lên quỹ đạo cũ — đúng bệnh real60.
- `llm_calls.sqlite` cũ: `ALTER TABLE` **chỉ ở write path**, guard `PRAGMA table_info`.
  Read path (`telemetry`, `verify`) dùng `COALESCE(superseded,0)=0`, **không bao giờ ALTER**.
- `data/runs/real60_spatial/` + `mock60_spatial/`: **KHÔNG sửa một byte** (`git status` rỗng).
- `World.luu_checkpoint(dir)` giữ nguyên chữ ký/hành vi; journal là **wrapper additive** ở `run.py`.

## 7. Claim boundary (BẮT BUỘC)

- **Được nói:** artifact của một run bị ngắt & resume **tái lập được bit-for-bit từ transcript, không
  mạng**; journal liên tục/duy nhất/khớp checkpoint; `call_burned` là chi phí thật, `call_effective`
  là số call trên quỹ đạo.
- **"Resume ⇒ cùng `world_hash` như run liền" CHỈ đúng khi provider là HÀM THUẦN của prompt**
  (FakeTransport). **LLM thật KHÔNG thuần** (temperature>0, batching phía server). Với run real,
  tính chất chứng minh được là **"artifact của run đã resume TỰ NHẤT QUÁN và replay ra CHÍNH HASH
  CỦA NÓ"** — KHÔNG phải "resume == liền mạch". Test matrix có **CẢ HAI** biến thể

### A-03 — exit code nghe theo nhãn, không nghe theo `failed()` (BLOCKING)

### A-11 — `cost_accounting_split` là tautology ⇒ identity THẬT

`burned = COUNT(*)`, `effective = COUNT(superseded=0)` ⇒ `burned >= effective` **luôn đúng theo
định nghĩa** (vẫn PASS cả sau khi DELETE row), mà vẫn được in như bằng chứng ở mọi block PASS.
Thay bằng `cost_accounting_identity` (`tools/verify_research_run.py:284-303`, hard), ba vế:

1. `call_burned == call_effective + superseded` (phân hoạch kín);
2. `superseded == Σ journal_recovery.rows_superseded` — **sổ đối ứng**: mỗi lần resume tuyên bố
   nó vừa đánh dấu bao nhiêu row; cột `superseded` phải khớp con số đã tuyên bố;
3. `MAX(call_id) == COUNT(*)` — `call_id` là AUTOINCREMENT (`minds/gateway.py:57`) ⇒ max == count
   ⟺ **chưa từng có row nào bị xóa**. Đây là vế duy nhất bắt được "xóa row cho chi phí đẹp".

`p0gate_resume`: `call_burned=1115 = call_effective=920 + superseded=195; Σrecovery=195;
MAX(call_id)=1115`. Negative test: `tests/test_resume_journal.py::test_a11_xoa_row_llm_calls_bi_bat`
xóa 3 row ⇒ khẳng định check CŨ vẫn PASS (tiền đề) rồi khẳng định check MỚI FAIL.
